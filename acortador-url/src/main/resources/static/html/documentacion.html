DESARROLLO DE PROYECTO ACORTADOR-URLs:

1.Introducción:

En el presente documento se expone toda la información relativa al desarrollo del proyecto Acortador-Url. Tiene principalmente una orientación muy práctica en cuanto el hilo conductor del documento son las distintas fases del desarrollo del mencionado proyecto.

3. Análisis Funcional:

La funcionalidad solicitada para este proyecto es básicamente poder convertir una url larga, de una longitud variable, en una url corta que podría ser por ejemplo un identificador. Además se debe incorporar un cliente web para el uso de estos servicios.  A partir del análisis de estos requisitos esenciales se crean los siguientes requisitos funcionales específicos:

  • Crear una funcionalidad o servicio que se le pase como entrada una url larga y devuelva una url corta, almacenando la relación url larga y url corta en una tabla o fichero si estos no están ya almacenados.
  • Crear un servicio que se le pase como entrada una url corta y devuelva una larga.
  • Crear un servicio que se le pase como entrada una url corta y haga la redirección a la larga.
  • Crear una funcionalidad que transforme una url larga en una corta, aplicando una transformación generar un identificador que relacione a ambos de manera unívoca.
  • El almacenamiento debe ser en tabla o fichero para no perder el histórico cada si se reinicia el servicio. Además se utilizará memoria dinámica (tabla Hash) para la gestión de los datos de modo más eficiente en tiempo de ejecución.
  • Creación de un cliente web que pueda solicitar una url corta a partir de una larga. Y viceversa.  Además se incluirá la funcionalidad de que cada vez que solicite la url larga a partir de la corta se le rediccionará a la dirección larga real.

4. Diseño Arquitectura:

Según los requisitos solicitados por el cliente este proyecto encaja como aplicación WEB. Antes de seleccionar la tecnología para llevarlo a cabo, se fijará la arquitectura de dicha aplicación.

Se elige una arquitectura MVC (modelo-vista-controlador) porque esta estructura ayudará enormemente en las fases de desarrollo permite desarrollar cada capa de manera independiente, pero sobre todo en la fase de mantenimiento ante errores o modificaciones de funcionalidad es mucho más eficaz. En nuestro caso:

  • Modelo: Contendrá el almacen tanto físico, como dinámico, y las estructuras de datos (beans) que se manejen en la aplicación.
  • Controlador: Contendrá todos los servicios que permiten ofrecer la funcionalidad requerida, siendo estos servicios invocados desde la Vista y por otro lado, desde el controlador se accederá al modelo para obtención y almacenamiento de datos.
  • Vista: Es la interfaz que se ofrecerá al usuario para que haga uso de los servicios que se ofrecen desde el controlador.

4.1 Tecnologías

Hay muchas posibilidades tecnológicas para llevar a cabo esta aplicación: Java JEE, Java con Spring, .Net, Nodejs, etc. Para seleccionar la mejor tecnología a utilizar cuenta mucho el conocimiento que se tenga de dicha tecnología siempre y cuando se adapte razonablemente, y también la estabilidad de dicha tecnología en el momento de seleccionarla. También las prestaciones necesarias de la aplicación es una cosa a tener muy en cuenta.

Para este proyecto se ha seleccionado Java con Spring por razones de  rapidez de desarrollo, pero sobre todo por  claridad y mantenimiento posterior ya que su estructura es muy visual una vez que se tienen los conocimiento mínimos.

Se desarrollará una estructura MCV integrada en el mismo proyecto. Aunque es muy común en diverso proyectos desarrollar las capas de Modelo y Controlador por un lado y en otro proyecto la parte de la Vista, usando en cada una de estas dos partes las tecnología que mejor se adecuen.

5. Pruebas:

Se crearán un set de pruebas automáticas con Test Junit de los servicios que se ejecutarán a petición del desarrollador y también podrán ser integradas en las tareas que se realizan en las fases de compilación, integración e implantación con herramientas tipo Jenkins.

  • Test1: Obtener url corta a partir de url larga: Deberá devolver una url corta que deberá coincidir con el valor que ya se ha preparado de antemano.
  • Test2: Obtener url larga a partir de la corta: Deberá devolver a partir de la url corta anterior la url larga anterior.
  • Test3: Chequear la redirección de url corta: Deberá redireccionar la url larga, se chequeara si lo hace correctamente.

6. Implantación y Despliegue:

Este proyecto genera un fichero .war que es desplegable en un servidor de aplicaciones Jboss o en servidores web tomcat. Las dos formas para el depliegue son:

  • Colocar el fichero acortador-url-1.0.0.war en el directorio de despliegue de jboss o tomcat y reiniciar el servidor.
  • Ejecutar desde línea de comandos donde esté el fichero .war → java -jar acortador-url-1.0.0.war

Con esto quedan arrancados los servicios y son accesibles desde un navegador con la url → http://localhost:8080/inicio

Configuración de Trazas:

  • En el fichero log4j2.xml están configuradas actualmente las trazas en el directorio /tmp/log. Aquí se genera un fichero rotativo cuando llega al tamaño de 2Mb que contiene el nombre de la aplicación más la fecha de creación.

Configuración de Almacén:

  • El fichero donde se almacena el histórico de urls accedidas se encuentra en /tmp/almacenUrls.csv. Es necesario que exista el directorio /tmp, en cuanto al fichero si no existe lo crea. La ruta de este fichero está definida en el fichero application.properties.
